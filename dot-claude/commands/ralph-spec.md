## Phase 0 — 작업 유형 선택

> 어떤 작업이야?
> A) 새 프로젝트  B) 기능 추가  C) 기능 변경  D) 버그 픽스

---

## [A] 새 프로젝트

1. 어떤 프로젝트인지, 해결하려는 문제, 주요 사용자, 성공 기준을 한 번에 하나씩 물어봐.
2. JTBD 정리 → 확인 후 Topics of Concern 분해 (and 없이 한 문장 테스트).
3. 기술 스택 결정 (경험 수준 / 선호 스택 / 배포 환경 / 규모).
4. 각 Topic별 엣지 케이스 · 우선순위 · 비목표 · 성능 제약 물어봐.
5. → **Spec 작성**

---

## [B] 기능 추가

1. 추가할 기능, 필요한 이유, 기존 기능과의 연결점, 성공 기준을 한 번에 하나씩 물어봐.
2. 기존 `specs/*` 와 관련 소스를 탐색해서 — 연결될 모듈, 이미 구현된 부분, 기존 패턴 파악. 요약 후 확인.
3. Topics of Concern 분해.
4. 각 Topic별 엣지 케이스 · 기존 기능과의 충돌 · 비목표 · 성능 제약 물어봐.
5. → **Spec 작성**

---

## [C] 기능 변경

1. 바꾸고 싶은 기능, 지금 동작의 문제점, 기대 동작, 변경 불가 영역을 한 번에 하나씩 물어봐.
2. 기존 `specs/*` 와 관련 소스를 탐색해서 — 현재 구현 위치, 영향 범위 파악. 요약 후 확인.
3. 엣지 케이스 · 회귀 위험 · 비목표 · 완료 기준 물어봐.
4. → **Spec 작성**

---

## [D] 버그 픽스

1. 발생 문제, 현재/기대 동작, 재현 조건을 한 번에 하나씩 물어봐.
2. 관련 소스와 `specs/*` 를 탐색해서 근본 원인(root cause) 파악. 아래 형식으로 요약 후 확인:
   ```
   근본 원인 / 영향 범위 / 관련 파일:라인 / 기존 spec 유무 / 기존 테스트 유무
   ```
3. 엣지 케이스 · 회귀 위험 · 비목표 · 완료 기준 물어봐.
4. → **Spec 작성**

---

## Spec 작성 (공통)

작성할 파일 목록 먼저 보여주고, 확인 후 작성.

- [A] `specs/overview.md` + `specs/[topic].md`
- [B] `specs/[feature].md` 신규 (기존 파일과 겹치면 수정)
- [C] 기존 `specs/[feature].md` 수정
- [D] 기존 `specs/[feature].md` 수정 (없으면 `specs/fix-[topic].md`)

[A][B]는 표준 spec 형식. [C][D]는 **Current Behavior / Expected Behavior / Regression Guard** 섹션을 추가.

---

## Spec 검증 (자동 실행)

작성 즉시 5가지 검증: 완전성 · 일관성 · 실현 가능성 · 테스트 가능성 · 의존성.
문제 항목은 승인 없이 바로 수정. 완료 후 파일 목록과 acceptance criteria 개수 요약.

```
# 터미널 방식
./loop.sh plan  # IMPLEMENTATION_PLAN.md 갱신
./loop.sh       # 빌드 루프 시작

# Claude Code 내 방식 (컨텍스트 소모 없음)
/ralph-plan     # IMPLEMENTATION_PLAN.md 생성 (1회)
/ralph-loop     # 빌드 시작 (5회)
```

## Phase 1 — 프로젝트 목표 파악

아래 질문들을 한 번에 하나씩 순서대로 물어봐:
1. 어떤 프로젝트를 만들고 싶은지 자유롭게 설명해줘.
2. 이 프로젝트로 해결하려는 핵심 문제가 뭐야? (사용자 관점에서)
3. 주요 사용자는 누구야?
4. 성공했다고 느끼려면 어떤 상태가 되어야 해?

답변을 바탕으로 JTBD 목록을 정리해서 보여줘. 내가 확인(또는 수정)하면 다음으로 넘어갈 것.

## Phase 2 — Topics of Concern 분해

JTBD를 Topics of Concern으로 분해해서 보여줘.
각 Topic이 "한 문장에 and 없이 설명 가능한가" 테스트를 통과했는지 확인.
내가 확인하면 다음으로 넘어갈 것.

## Phase 3 — 기술 스택 결정

아래 질문들을 한 번에 하나씩 물어봐:
1. 개발 경험 수준은? (프론트/백엔드 각각)
2. 선호하는 언어나 프레임워크가 있어?
3. 배포 환경은? (Vercel, AWS, 자체 서버 등)
4. 예상 사용자 규모와 데이터 양은?

답변을 바탕으로 기술 스택을 추천하고 이유를 설명해줘. 내가 확인하면 다음으로 넘어갈 것.

## Phase 4 — 빈틈 메우기

각 Topic of Concern에 대해 아래를 한 번에 하나씩 물어봐:
- 엣지 케이스: 예외 상황이나 실패 케이스가 있어?
- 우선순위: MVP에 꼭 필요한 것과 나중에 추가해도 되는 것은?
- 비목표: 명시적으로 이번에 안 만드는 것은?
- 성능/제약: 속도, 용량, 보안 등 구체적인 제약이 있어?

## Phase 5 — Specs 작성

작성할 파일 목록을 먼저 보여줘. 내가 승인하면 작성 시작.

각 spec 파일은 아래 형식을 따를 것:
- **Overview** — 이 기능이 무엇인지 한 문단
- **Acceptance Criteria** — 번호 목록. 각 항목은 반드시:
  - 동사로 시작 ("사용자는 ~할 수 있다", "시스템은 ~해야 한다")
  - 구체적인 수치나 조건 포함 ("5초 이내", "10MB 이하" 등)
  - 테스트로 검증 가능한 형태
  - 구현 방법 언급 금지 (what만, how는 금지)
- **Edge Cases** — 예외 상황과 각각의 기대 동작
- **Non-goals** — 이 spec에서 다루지 않는 것

## Phase 6 — Spec 검증 (자동 실행)

specs 작성이 완료되면 즉시 아래 5가지 관점으로 전체 specs를 분석해줘. 사람의 입력 없이 자동으로 진행할 것.

**검증 항목:**

1. **모호함 제거** — acceptance criteria 중 수치/조건이 없는 항목 목록화
   - 예: "빠르게 로드된다" → "3초 이내에 로드된다"로 수정 필요
   - 예: "적절한 에러 메시지" → 구체적인 메시지 내용/형식 필요

2. **기술적 실현 가능성** — 선택한 기술 스택으로 구현 불가능하거나 과도한 항목
   - 성능 요구사항이 현실적인가
   - 선택한 스택의 한계를 벗어나지 않는가

3. **완전성** — 빠진 기능, 정의되지 않은 동작
   - 각 JTBD가 spec으로 완전히 커버되는가
   - 인증/에러처리/로딩상태 등 공통 관심사가 정의됐는가

4. **일관성** — spec 간 모순, 용어 불일치
   - 같은 개념을 다른 용어로 쓰는 경우
   - spec A와 spec B가 서로 충돌하는 경우

5. **의존성** — 구현 순서상 문제
   - 다른 spec에 의존하는 항목이 명시됐는가
   - 순환 의존이 있는가

검증 결과를 아래 형식으로 보여줄 것:

```
## Spec Validation Report

### 🔴 수정 필요 (모호하거나 불가능한 항목)
- [파일명] [항목]: 문제 설명 → 수정 제안

### 🟡 검토 권장
- [파일명] [항목]: 이유

### ✅ 통과
```

수정이 필요한 항목이 있으면 내 승인 없이 바로 수정하고, 수정 내용을 보고해줘.
수정 완료 후 최종 spec 파일 목록과 각 파일의 acceptance criteria 개수를 요약해줘.
EOFEEOF
